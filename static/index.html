🏗️ System Overview - How It All Works

        

          
Sprint Intelligence Multi-Agent Platform

          

            This platform combines orchestration, data engineering, retrieval, and LLM inference into one
            end-to-end workflow for sprint health intelligence. It is designed to support both interactive
            analysis and organization-scale ingestion pipelines.
          

          
How the System Works

          

            The Core Flow: Users submit a sprint (via JSON or natural language query) → The system orchestrates 11 specialized agents to
            analyze repository metadata, dependencies, issues, PRs, and commits → Agents build multi-modal feature vectors → RAG retrieval
            enriches context from ChromaDB → LLM reasoning (via Ollama) estimates completion probability and health status → Risk analysis and
            recommendations are generated → A narrative explanation is delivered through the dashboard.
          

          

            Key Design Principles:
          

          
    
            
  • Modular Agents: Each agent has a single responsibility (collect data, extract features, reason, explain).
    • 
            
  • Local-First: All inference runs locally via Ollama—no external LLM calls, full data privacy.
    • 
            
  • RAG-Grounded: Reasoning is constrained by vector-retrieved evidence from the indexed repository.
    • 
            
  • Async-Aware: Large ingestion jobs run in background threads with job IDs and polling APIs.
    • 
            
  • Explainable: Every output includes evidence traces, risk categories, and narrative justification.
    •

📦 Data Pipeline - Load & Processing Details

        

          
How Data Flows Through the System

          

            The data pipeline handles both on-demand ingestion (single sprint analysis) and bulk ingestion
            (entire organizations). It transforms raw GitHub data into indexed, searchable knowledge.
          

        


        
The 5-Stage Pipeline

        

          1. Ingest
          2. Normalize
          3. Enrich
          4. Embed & Index
          5. Query
        


        

          
Stage 1: Ingest

          

            What happens: The system fetches repository data via GitHub API (or local repo cloning for org pipelines).
            Data includes issues, pull requests, commits, branches, releases, and contributor activity. For organization-level ingestion,
            the system can clone/update multiple repos in parallel.
          

          
How: Uses GitHub REST API for issues/PRs, git CLI for commit history and file changes.
          Handles pagination (1000s of issues) and rate limiting transparently.

        


        

          
Stage 2: Normalize

          

            What happens: Raw GitHub data is parsed into consistent schemas. Issues become "Issue" objects,
            PRs become "PR" objects, and commits become "Commit" objects. Sprint boundaries are applied (group by sprint ID or date ranges).
          

          
Output: Normalized JSON with uniform structure regardless of GitHub API quirks.

        


        

          
Stage 3: Enrich

          

            What happens: The system adds derived data: dependency links, sentiment analysis of PR comments,
            code complexity metrics, team velocity calculations, and synthetic sprint generations (for model training).
          

          
Example: If issue #5 mentions issue #3, a "blocks" relationship is added.
          If PR #10 is linked to 5 issues, those connections are indexed for later retrieval.

        


        

          
Stage 4: Embed & Index

          

            What happens: Enriched documents (issue summaries, PR descriptions, commit messages, sprint overviews)
            are vectorized using an embedding model. Each document is converted to a vector in high-dimensional space (e.g., 1536 dimensions).
          

          
Indexed in: ChromaDB with per-repository collections. This enables fast
          semantic search ("Find sprints similar to current one") during inference.

        


        

          
Stage 5: Query & Retrieve

          

            What happens: During inference, the Reasoner Agent queries ChromaDB with the current sprint's feature vector.
            It retrieves the K most similar sprints with their outcomes and metadata. This evidence constrains LLM reasoning.
          

          
Result: LLM has real, grounded examples to reason from instead of pure hallucination.

        


        
Data Load & Performance

        
    
          
  • Single Sprint Ingestion: ~100 issues/PRs → loads in <2 seconds (on modern hardware).
    • 
          
  • Organization Ingestion: 50+ repos, 1000s of sprints → runs asynchronously with job tracking, typically 30-120+ seconds depending on GitHub rate limits.
    • 
          
  • Memory Footprint: Streaming ingestion to avoid loading entire org into RAM; documents are indexed directly into ChromaDB.
    • 
          
  • Scalability: ChromaDB is file-backed (SQLite), suitable for 1000s–10,000+ documents. For larger scale, consider vector DB with cloud backend.
    •

⚡ Inference Lifecycle - Complete Process

        

          
How a Request is Processed End-to-End

          

            When you submit a sprint for analysis, here's exactly what happens at each step:
          

        


        

          

            
📥 A. Request Intake (0-100ms)

            

              FastAPI endpoint receives JSON or natural language query. Validates sprint schema, repository exists,
              and user has permissions. Router Agent normalizes input into internal state representation.
              Rejected if malformed; returns 400 with error details.
            

          

          

            
🔍 B. Context Building (100-500ms)

            

              Collector Agent fetches fresh GitHub data (issues, PRs, commits). Feature Engineer derives metrics
              (velocity, burn-down, code churn, team patterns). RAG Reasoner queries ChromaDB for 3-5 most similar sprints
              with historical outcomes. Result: rich feature vectors + retrieved evidence context.
            

          

          

            
🧠 C. Model Reasoning (500-1500ms)

            

              Ollama LLM receives prompt: "Given these sprint metrics and these similar historical sprints,
              estimate completion probability." LLM reasons over the constrained context and produces structured output:
              completion %, health status, confidence score. All reasoning stays local (no external API calls).
            

          

          

            
⚠️ D. Post-Processing (1500-2000ms)

            

              Risk Assessor identifies red flags (blocker issues, team availability, dependencies).
              Recommender suggests interventions. Explainer generates markdown narrative with references to evidence:
              "Completion risk is HIGH because: [3 specific issues blocking], [2 PRs unreviewed], [team is at 110% capacity]."
            

          

          

            
📊 E. Delivery (2000-2100ms)

            

              Dashboard renders response: completion meter, risk cards, dependency graph, recommendation list,
              evidence timeline. All data is cached in browser session state. User can inspect evidence traces,
              drill into specific issues/PRs, and view explainability breakdown. JSON response is also available via API.

Key Points About Inference

        
    
          
  • Fast: Typical end-to-end inference is 2-3 seconds (cached ChromaDB + local LLM).
    • 
          
  • Local: All reasoning happens on your machine via Ollama. No data sent to external APIs.
    • 
          
  • Evidence-Based: Every output is grounded in retrieved historical examples and current sprint metrics.
    • 
          
  • Explainable: Every recommendation includes "why"—which risk triggered it, which evidence supports it.
    • 
          
  • Non-Blocking: Can handle multiple concurrent requests via FastAPI async handling.
    •

📚 Research Alignment & Design Goals

          
    
            
  • Explainability: Every output includes narrative reasoning, specific risk IDs, evidence traces (issue #5 blocks issue #3), and grounding in historical examples.
    • 
            
  • Scalability: Async org ingestion with UUID job tracking enables background processing of 100s of repos without blocking UI. Streaming data load avoids memory blowup.
    • 
            
  • Robustness: Strict mode (fail-fast on any data error) vs. resilient mode (graceful fallback, skip missing data). Both produce valid outputs.
    • 
            
  • Reproducibility: All run artifacts (metrics JSON, input sprint, retrieved evidence, LLM prompt, response) are saved for audit trails and benchmarking.
    • 
            
  • RAG Fidelity: Repository-scoped ChromaDB collections prevent evidence leakage from unrelated repos. Semantic search ensures retrieved examples are truly similar.
    • 
            
  • Privacy: 100% local inference—data never leaves your machine. No cloud calls, no telemetry, no data escapes.

Quick Start - 3 Steps

        
    
          
  1. 
            Load or Ingest Data:
            
      
              
    • Single Sprint: Click "Ingestion" → Enter repo name (e.g., badges/shields) → Submit

      • 
            
    
          
    2. 
          
  2.

      •

      •

      • 
            
    
          
    3. 
          
  3. 
            View Results:

      •

      •

      •

      • 
            
    
          
    4. 
        


        
Advanced: Understanding Outputs

        
    
          
  • Completion %: Estimated probability sprint finishes on time. 95-100% = high confidence, <70% = high risk.
    • 
          
  • Health Score: Team capacity, velocity stability, dependency complexity rolled into one metric (0-100).
    • 
          
  • Risks: Categorized by type (blocker issue, PR delay, capacity overload, etc.) with impact scores.
    • 
          
  • Recommendations: Specific actions (reassign task to person X, cut feature Y, unblock PR #Z) with estimated impact on completion %.
    • 
          
  • Evidence: Clickable links to specific GitHub issues/PRs that triggered each finding.
